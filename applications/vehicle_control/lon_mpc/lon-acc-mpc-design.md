# LonMPC 纵向加速度 MPC 设计文档

## 1. 概述

### 1.1 定位

`LonMPC` 是一个独立的纵向控制 MPC 模块，输出期望加速度 `a_des`，可移植到不同项目/平台。作为上层加速度规划器使用，下游负责 acc→torque 映射。

### 1.2 设计目标

- 独立模块，无框架耦合
- 输出期望加速度，不直接控制执行器
- 接口简洁，参数可配置
- 使用 HPIPM 求解器（stage-wise OCP QP）

---

## 2. 数学建模

### 2.1 模型 A：纯三阶积分器（主方案）

**状态**：`x = [s, v, a]ᵀ`（位移、速度、加速度）
**控制**：`u = j`（jerk）
**输出**：`a_des = x[2] = a`（无指令/实际区分，直接输出）

**连续时间**：

```
ṡ = v
v̇ = a
ȧ = j
```

**精确离散化**（dt = 0.1s）：

```
x(k+1) = Ad · x(k) + Bd · u(k)

Ad = [1   dt   0.5·dt²]     Bd = [(1/6)·dt³]
     [0   1    dt      ]          [0.5·dt²  ]
     [0   0    1       ]          [dt        ]
```

### 2.2 模型 B：带惯性环节（扩展方案）

**状态**：`x = [s, v, a, a_cmd]ᵀ`
**控制**：`u = j`（jerk）
**输出**：`a_des = x[3] = a_cmd`（指令加速度）

**输出语义说明**：

模型 B 中 `a`（x[2]）和 `a_cmd`（x[3]）含义不同：

| 状态 | 含义 | 说明 |
|------|------|------|
| `x[2] = a` | 模型预测的实际加速度 | 经惯性滤波后的值 |
| `x[3] = a_cmd` | 指令加速度 | MPC 规划发出的期望值 |

选择输出 `a_cmd` 的理由：作为上层规划器，下发给底层的应该是**指令**。惯性环节的作用是让 MPC 在规划时感知下游执行延迟并提前给指令，但最终输出仍是指令本身，下游直接执行即可，无需再做逆模型补偿。

**连续时间**：

```
ṡ = v
v̇ = a
ȧ = (a_cmd - a) / τ
ȧ_cmd = j
```

**状态空间**：

```
A = [0  1  0      0  ]     B = [0]
    [0  0  1      0  ]         [0]
    [0  0  -1/τ   1/τ]         [0]
    [0  0  0      0  ]         [1]
```

离散化需用 ZOH 或双线性变换（前向欧拉在 dt > τ 时不稳定）。

**注意**：两套模型不可通过 τ→0 统一，需独立实现。详见附录 A.3。

### 2.3 代价函数

```
J = Σ(k=0 to N-1) [ 0.5·(x_k - x_ref_k)ᵀ Q (x_k - x_ref_k) + 0.5·u_kᵀ R u_k ]
    + 0.5·(x_N - x_ref_N)ᵀ QN (x_N - x_ref_N)
    + Σ(k=1 to N) [ 0.5·ρ_v · ε_v_k² + 0.5·ρ_a · ε_a_k² ]
```

注意：
- 包含 0.5 系数，与 HPIPM 标准形式 `0.5·xᵀQ_kx + q_kᵀx` 直接对应
- 松弛变量从 k=1 开始（k=0 为初始状态等式约束，无需软约束）

参数说明：
- `Q = diag(q_s, q_v, q_a)` — 状态跟踪权重
- `QN = diag(q_s_N, q_v_N, q_a_N)` — 终端代价权重
- `R = [q_jerk]` — jerk 惩罚（舒适性）
- `ρ_v` — 速度松弛惩罚权重
- `ρ_a` — 加速度松弛惩罚权重
- `ε_v_k, ε_a_k` — 松弛变量（≥ 0）

`ρ_v`、`ρ_a` 通常比 Q 大 1~2 个数量级，确保"宁可跟踪误差大，也尽量不违反约束"。

### 2.4 约束

**硬约束**（不可违反）：
```
j_min ≤ u_k ≤ j_max
```

**软约束**（通过松弛变量，允许轻微违反以保证有解）：
```
v_min - ε_v_k ≤ v_k ≤ v_max + ε_v_k,   ε_v_k ≥ 0
a_min - ε_a_k ≤ a_k ≤ a_max + ε_a_k,   ε_a_k ≥ 0
```

### 2.5 MPC 参数

| 参数 | 值 |
|------|-----|
| 预测步长 N | 20 |
| 采样时间 dt | 0.1s |
| 预测时域 | 2.0s |
| 状态维度 nx | 3（主方案）/ 4（扩展方案） |
| 控制维度 nu | 1 |

N=20（2.0s）的选择理由：作为上层加速度规划器，不需要像 torque MPC（N=30, 3.0s）看那么远；2s 预测时域覆盖大多数工况（城市跟车、正常加减速），极端工况（高速急刹）由松弛变量兜底。

### 2.6 坐标系与时序约定

**位移 s 坐标系**：采用**相对坐标**，每个控制周期 `s0 = 0`，参考轨迹 `s_ref[k]` 表示相对于当前位置的纵向位移。避免全局里程累积导致数值精度下降。

**参考轨迹时间对齐**：
- `x_ref[0]` 对应当前时刻 t（与初始状态 `[s0, v0, a0]` 对应同一时刻）
- `x_ref[k]` 对应 `t + k·dt`
- 若上游规划器 dt 与 MPC dt 不同，调用方需在 `Update()` 前完成插值对齐

**输出 `a_des` 时序语义**：
- `a_des = x_pred[1]`，即 MPC 预测的 **t+dt 时刻加速度**
- 物理含义：在当前控制周期内，以 `jerk_des` 作用 dt 后应达到的目标加速度
- 下游执行器应在下一个控制周期开始时达到此加速度值

**输入校验要求**：
- `s_ref, v_ref, a_ref` 长度必须为 N+1，否则返回 solver_status = -1
- 约束必须满足 `j_min < j_max`, `a_min < a_max`, `v_min < v_max`

## 3. 模块架构

### 3.1 结构

```
┌──────────────────────────────────────────────────────┐
│                     LonMPC                            │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │  LonMPCConfig（初始化参数）                     │  │
│  │  - N, dt                                       │  │
│  │  - Q, QN, R, ρ_v, ρ_a                         │  │
│  │  - 约束边界 (j_min/max, v_min/max, a_min/max) │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │  ModelBuilder（建模层）                         │  │
│  │  - 计算 Ad, Bd（初始化时一次）                  │  │
│  │  - 每周期构建各 stage 的 q_k, r_k              │  │
│  │  - 构建约束边界                                │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │  HpipmInterface（求解器接口层）                 │  │
│  │  - 管理 HPIPM 内存分配                         │  │
│  │  - 填充 OCP QP 数据结构                        │  │
│  │  - 调用 IPM 求解                               │  │
│  │  - 提取解                                      │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 3.2 接口定义

```cpp
struct LonMPCInput {
  // 初始状态
  double s0;
  double v0;
  double a0;

  // 参考轨迹 (长度 N+1)
  std::vector<double> s_ref;
  std::vector<double> v_ref;
  std::vector<double> a_ref;

  // 约束 (可选运行时覆盖，否则用初始化默认值)
  double v_min;
  double v_max;
  double a_min;
  double a_max;
  double j_min;
  double j_max;
};

struct LonMPCOutput {
  // 控制指令
  double a_des;           // 期望加速度（核心输出）
  double jerk_des;        // 对应的 jerk

  // 预测序列 (长度 N+1)
  std::vector<double> s_pred;
  std::vector<double> v_pred;
  std::vector<double> a_pred;
  // 控制序列 (长度 N)
  std::vector<double> jerk_pred;

  // 求解信息
  int solver_status;      // 0=成功, 其他=失败码
  int iter_count;         // IPM 迭代次数
  double max_residual;    // KKT 残差（解质量指标，越小越好，见下表）
  double solve_time_ms;   // 求解耗时
};

class LonMPC {
public:
  void Init(const LonMPCConfig& config);
  LonMPCOutput Update(const LonMPCInput& input);
  void Reset();
};
```

### 3.3 调用流程

```
每个控制周期：
  1. 上层填充 LonMPCInput（初始状态 + 参考轨迹 + 约束）
  2. 调用 Update()
     ├── ModelBuilder: 用 x_ref 计算梯度项 q_k = -Q · x_ref_k
     ├── HpipmInterface: 填充 HPIPM 数据 → 求解
     └── 解析 solution → 填充 LonMPCOutput
  3. 取 LonMPCOutput.a_des 下发给底层
```

## 4. HPIPM 集成

### 4.1 问题映射

HPIPM OCP QP 标准形式：

```
min  Σ(k=0 to N-1) [ 0.5·x_kᵀ Q_k x_k + q_kᵀ x_k
                    + 0.5·u_kᵀ R_k u_k + r_kᵀ u_k ]
     + 0.5·x_Nᵀ Q_N x_N + q_Nᵀ x_N

s.t. x(k+1) = A_k x_k + B_k u_k + b_k
     lbx_k ≤ x_k ≤ ubx_k
     lbu_k ≤ u_k ≤ ubu_k
```

映射：

**动力学**（每个 stage 相同）：
```
A_k = Ad,  B_k = Bd,  b_k = 0
S_k = 0  (无状态-控制交叉项)
```

**初始状态约束**：
```
stage 0: lbx_0 = ubx_0 = [s0, v0, a0]  (等式约束，固定初始状态)
stage 0: ns = 0  (无软约束，因为状态已完全确定)
```

**代价**：

展开 `0.5·(x-x_ref)ᵀQ(x-x_ref) = 0.5·xᵀQx - xᵀQ·x_ref + const`，映射到 HPIPM `0.5·xᵀQ_kx + q_kᵀx`：
```
Q_k = Q = diag(q_s, q_v, q_a)          (直接对应，0.5已匹配)
q_k = -Q · x_ref_k                      (每步更新)
R_k = R = [q_jerk]
r_k = [0]
Q_N = QN = diag(q_s_N, q_v_N, q_a_N)
q_N = -QN · x_ref_N
```

（常数项 `0.5·x_ref_kᵀ Q x_ref_k` 不影响最优解，丢弃；输出 cost_value 不含此常数偏移）

**约束**（stage k=1 to N）：
```
lbu_k = [j_min],  ubu_k = [j_max]       (硬约束)
lbx_k = [无约束, v_min, a_min]           (soft, k≥1)
ubx_k = [无约束, v_max, a_max]           (soft, k≥1)
```

### 4.2 Soft Constraint 配置

HPIPM 原生支持 soft box constraint。配置如下（仅 stage k=1 到 N，stage 0 无软约束）：

```
ns_vec[0] = 0              // stage 0 无 soft (初始状态固定)
ns_vec[k] = 2, k=1..N     // 2 个 soft 约束 (v 和 a)

soft_idx_x = [1, 2]       // 第1个(v)和第2个(a) box constraint 是 soft

// 速度松弛（对应 HPIPM: 0.5·Zl·sl² + zl·sl）
Zl[0] = Zu[0] = ρ_v      // 二次惩罚权重（HPIPM 内含 0.5，与代价函数 0.5·ρ_v·ε² 匹配）
zl[0] = zu[0] = 0         // 无线性惩罚

// 加速度松弛
Zl[1] = Zu[1] = ρ_a
zl[1] = zu[1] = 0
```

HPIPM 内部自动管理松弛变量的引入和求解。

### 4.3 内存管理

HPIPM 使用预分配内存模式（无运行时 malloc）：

```cpp
// 初始化阶段（Init 中一次）:
1. d_ocp_qp_dim     — 设置问题维度 (N, nx, nu, ns...)
2. d_ocp_qp         — 分配 QP 数据结构
3. d_ocp_qp_sol     — 分配 solution 结构
4. d_ocp_qp_ipm_arg — 设置 IPM 参数
5. d_ocp_qp_ipm_ws  — 分配 IPM workspace

// 运行阶段（每个 Update 周期）:
1. 填充 q_k, 约束边界（A_k, B_k, Q_k, R_k 不变无需重填）
2. d_ocp_qp_ipm_solve()
3. 从 d_ocp_qp_sol 取出 x*, u*
```

### 4.4 求解参数

| 参数 | 值 |
|------|-----|
| IPM 容差 | tol = 1e-6（小规模问题开销可忽略；若实时性不满足可放松至 1e-4） |
| 最大迭代次数 | max_iter = 50（正常情况 5~15 次收敛，超过 30 次视为异常） |
| 初始 barrier | mu0 = 1e1 |
| warm-start | 支持（直接复用上一周期 HPIPM sol 结构） |

**Warm-start 策略**：采用直接复用方式（HPIPM `warm_start=1`），即上一周期的完整 solution 保留在 sol 结构中作为本周期的初始迭代点。不做时间平移（shift），理由：问题规模极小（nx=3, N=20），cold-start 也仅需 10~15 次迭代，shift 带来的收益有限。

**Cold-start 触发条件**：首次调用、`Reset()` 后首次调用、或参考轨迹突变（由调用方通过 `Reset()` 显式触发）时，使用 cold-start（HPIPM sol 结构已清零）。

### 4.5 求解失败处理（Fallback 策略）

**max_residual 解质量判定**：

| max_residual | 含义 | 处理 |
|-------------|------|------|
| < 1e-6 | 高精度收敛（与 tol 一致） | 正常使用 |
| 1e-6 ~ 1e-4 | 解质量良好 | 正常使用 |
| 1e-4 ~ 1e-2 | 解质量一般，可能在约束边界附近数值条件差 | 可使用，建议监控 |
| > 1e-2 | 解质量差，不应信任 | 应触发 fallback |

车辆控制场景下 **< 1e-4 即为合格**。若频繁出现 > 1e-4，说明 QP 存在数值病态（权重量级差异过大或约束互斥）。

**单次失败处理**：
- `solver_status != 0` 时，`a_des` 输出上一周期成功求解的 `a_des` 值（hold-last-acceleration）
- `jerk_des` 输出 0（不施加额外 jerk）
- 内部保存上一次成功的 `LonMPCOutput` 用于 fallback

**连续失败降级**：
- 内部维护失败计数器 `fail_counter`
- 连续失败超过 `max_fail_count`（默认 3 次）时，`solver_status` 标记为降级状态
- 调用方可通过 `solver_status` 判断是否切换到备份控制器（如 PID 速度环）

**输出保护**：
- 对 `a_des` 做 NaN/Inf 检查，异常时回退到 hold-last 或 0
- 连续求解成功时重置 `fail_counter`

**与上层状态机的交互**：
- LonMPC 本身不做模式切换，只通过 `solver_status` 上报状态
- 上层根据 `solver_status` 决定是否启用备份控制器（PID 等）
- 调用方在切回 MPC 前应调用 `Reset()` 触发 cold-start

## 5. 权重参数设置

### 5.1 当前框架参数参考

当前 `LonOcpControl`（torque 输出 MPC）使用的默认权重：

| 参数 | 当前框架值 | 含义 |
|------|-----------|------|
| q_s | 3.0 | 位移跟踪 |
| q_s_N | 3.0 | 位移终端代价 |
| q_v | 30.0 | 速度跟踪 |
| q_v_N | 30.0 | 速度终端代价 |
| q_jerk | 1.0 | jerk 惩罚 |
| q_r (control) | 2.0 | 控制量惩罚 |
| q_vel_bound (ρ_v) | 100.0 | 速度松弛惩罚 |
| q_slack_min_vel | 10.0 | 低速松弛 |
| q_slack_max_vel | 10.0 | 高速松弛 |

当前框架约束默认值：

| 参数 | 值 | 说明 |
|------|-----|------|
| jerk_min | -10.0 (驱动) / -max_brk_jerk (制动) | torque rate 域 |
| jerk_max | max_prop_jerk (驱动) / 10.0 (制动) | torque rate 域 |
| v_min | 0.0 m/s | 不允许倒车 |
| 预测步长 | 30 步 | N=30 |
| 采样频率 | 10 Hz | dt=0.1s |

### 5.2 LonMPC 推荐参数

我们的模块输出加速度、控制量是 jerk（m/s³），量纲与当前 torque 框架不同。基于当前框架的权重比例关系，推荐初始参数：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| q_s | 3.0 | 位移跟踪（沿用） |
| q_s_N | 3.0 | 位移终端 |
| q_v | 30.0 | 速度跟踪（主导权重） |
| q_v_N | 30.0 | 速度终端 |
| q_a | 1.0 | 加速度跟踪（新增，当前框架无此项） |
| q_a_N | 1.0 | 加速度终端 |
| q_jerk | 1.0 | jerk 惩罚（舒适性） |
| ρ_v | 100.0 | 速度松弛惩罚 |
| ρ_a | 100.0 | 加速度松弛惩罚 |
| j_min | -5.0 m/s³ | jerk 下界（制动方向，待标定） |
| j_max | 3.0 m/s³ | jerk 上界（驱动方向，待标定） |

注：jerk 非对称约束基于制动响应通常快于加速（工程经验），具体值需实车标定验证。

**模型 B 扩展参数**（仅模型 B 使用）：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| τ | 0.1~0.2s | 执行器惯性时间常数，需根据实际平台标定 |
| Q(4,4) | 同 q_a | 指令加速度权重 |
| v_min | 0.0 m/s | 速度下界 |
| v_max | 60.0 m/s | 速度上界 |
| a_min | -4.0 m/s² | 加速度下界（舒适制动） |
| a_max | 2.0 m/s² | 加速度上界（正常驱动） |

### 5.3 调参指导

**权重比例关系**：

```
q_v >> q_s > q_a ≈ q_jerk
```

- `q_v / q_s ≈ 10`：速度跟踪优先于位移跟踪
- `q_jerk` 越大越平滑但响应越慢
- `ρ_v, ρ_a` 应比 Q 大 1~2 个数量级

**各权重调整效果**：

| 权重 | 调大效果 | 调小效果 |
|------|---------|---------|
| q_s | 位移跟踪更紧，可能导致加速度波动 | 位移允许偏差更大 |
| q_v | 速度跟踪更紧（最重要） | 速度误差容忍度高 |
| q_a | 加速度更贴近参考（减小超调） | 加速度自由度大 |
| q_jerk | 更平滑舒适，响应慢 | 响应快但可能有抖动 |
| ρ_v, ρ_a | 约束越"硬"，越不愿违反 | 约束越"软"，紧急工况时允许越界 |

**工况调参策略**：

| 工况 | 策略 |
|------|------|
| 巡航 | 默认参数即可 |
| 停车 | q_s 增大 ×10，确保精确停到位 |
| 起步 | q_jerk 适当减小，允许较大 jerk 快速响应 |
| 急刹 | a_min 放宽至 -5~-6，ρ_a 适当减小允许越界 |
| 跟车 | q_v 为主导，q_s 适当增大保持车距 |

### A.1 时域与频域转换

| 方向 | 变换名称 | 作用 |
|------|---------|------|
| 时域 → 频域 | 拉普拉斯变换 (Laplace Transform) | 将微分方程转为代数方程 |
| 频域 → 时域 | 反拉普拉斯变换 (Inverse Laplace Transform) | 将传递函数转回微分方程 |

核心对应规则（零初始条件下）：

| 时域 | 频域（拉普拉斯域） |
|------|-------------------|
| `x(t)` | `X(s)` |
| `ẋ(t)` | `s·X(s)` |
| `ẍ(t)` | `s²·X(s)` |

### A.2 一阶惯性环节的三种等价表示

#### 微分方程 → 传递函数（拉普拉斯变换）

起点：
```
ȧ = (a_cmd - a) / τ
```

展开并做拉普拉斯变换：
```
τ · ȧ + a = a_cmd
τ · s · A(s) + A(s) = A_cmd(s)
A(s) · (τs + 1) = A_cmd(s)
```

得传递函数：
```
G(s) = A(s) / A_cmd(s) = 1 / (τs + 1)
```

#### 传递函数 → 微分方程（反拉普拉斯变换）

起点：
```
A(s) / A_cmd(s) = 1 / (τs + 1)
```

交叉相乘：
```
A(s) · (τs + 1) = A_cmd(s)
τ · s · A(s) + A(s) = A_cmd(s)
```

反拉普拉斯变换回时域（`s·A(s)` → `ȧ(t)`）：
```
τ · ȧ + a = a_cmd
ȧ = (a_cmd - a) / τ
```

#### 微分方程 → 状态空间矩阵

完整 4 状态系统 `ẋ = Ax + Bu`：
```
┌ ṡ     ┐   ┌ 0   1    0      0   ┐ ┌ s     ┐   ┌ 0 ┐
│ v̇     │   │ 0   0    1      0   │ │ v     │   │ 0 │
│ ȧ     │ = │ 0   0   -1/τ   1/τ  │ │ a     │ + │ 0 │ · u
└ ȧ_cmd ┘   └ 0   0    0      0   ┘ └ a_cmd ┘   └ 1 ┘
```

第三行展开即还原微分方程：
```
ȧ = (-1/τ)·a + (1/τ)·a_cmd = (a_cmd - a) / τ
```

三者关系：
```
微分方程  ←──拉普拉斯变换/反拉普拉斯变换──→  传递函数
    │
    └──按状态变量排列系数──→  状态空间矩阵
```

### A.3 惯性模型无延迟退化（奇异摄动降阶）

**问题**：能否通过令 τ→0 将 4 状态模型退化为 3 状态模型？

**频域**：可以。
```
G(s) = 1/(τs + 1) → τ=0 → G(s) = 1
```

**时域（状态空间）**：不可以。
```
ȧ = (-1/τ)·a + (1/τ)·a_cmd
```
τ→0 时，`±1/τ → ±∞`，矩阵数值奇异。

**正确做法**：当 τ→0，惯性动态无穷快，`a` 瞬间跟上 `a_cmd`：
```
τ→0  ⟹  a = a_cmd（代数约束）
```

消去 `a` 状态，系统降阶：
```
4阶: [s, v, a, a_cmd]   ȧ = (a_cmd - a)/τ
                ↓ τ→0, a ≡ a_cmd
3阶: [s, v, a]           ȧ = u (jerk)
```

**结论**：两套模型必须独立实现，不可通过参数 τ 统一。

### A.4 τ 与 dt 关系的数值稳定性分析

对于 dt = 0.1s 的系统，惯性时间常数 τ 的选取影响数值行为：

#### 前向欧拉离散化

离散极点 = `1 - dt/τ`，稳定性条件：`|1 - dt/τ| < 1` → **τ > dt/2**

| τ | dt/τ | 离散极点 | 状态 |
|---|------|---------|------|
| 0.2s | 0.5 | 0.5 | 稳定，响应适中 |
| 0.1s | 1.0 | 0.0 | 临界（deadbeat） |
| 0.05s | 2.0 | -1.0 | 临界振荡 |
| 0.03s | 3.3 | -2.3 | 发散 |

#### ZOH（精确离散化）

离散极点 = `e^{-dt/τ}`，理论上永远稳定，但有条件数问题：

| τ | dt/τ | 离散极点 | 矩阵条件数 |
|---|------|---------|-----------|
| 0.2s | 0.5 | 0.61 | 良好 |
| 0.1s | 1.0 | 0.37 | 良好 |
| 0.05s | 2.0 | 0.14 | 可接受 |
| 0.01s | 10 | 0.0000454 | 极差 |

#### 实用建议

对于 dt = 0.1s：

```
τ ≥ dt = 0.1s      → 安全，所有离散化方法都 OK
τ ∈ [dt/2, dt)     → 0.05~0.1s，ZOH/Bilinear 可用，前向欧拉勉强
τ < dt/2 = 0.05s   → 不推荐，近似"无延迟"但条件数差
```

- 想建模快速执行器：τ = 0.1s（= dt），一步跟踪 63%
- 想建模慢执行器：τ = 0.2~0.5s
- 想近似无延迟：直接用 3 状态模型，不要用小 τ 近似
