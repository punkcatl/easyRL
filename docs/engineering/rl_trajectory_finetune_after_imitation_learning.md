# RL-Based Trajectory Fine-Tuning After Imitation Learning

## 1. Background

End-to-end autonomous driving often starts with imitation learning. A base policy learns to map perception, routing, and history into a future ego trajectory.

```text
Input: sensors / BEV / Occupancy / map / navigation / history
Output: trajectory τ = {(x1,y1,v1,yaw1), ..., (xT,yT,vT,yawT)}
```

The behavior cloning objective is usually:

```text
L_BC = ||τ_pred - τ_expert||
```

This gives the model basic driving capability, but it has limitations: averaged behavior, distribution shift, short-horizon fitting, copying bad human habits, and a gap between open-loop error and closed-loop driving quality.

## 2. Why RL Fine-Tuning Is Used

After imitation learning, RL, offline RL, or preference learning can be used to optimize the consequences of executing a trajectory, rather than only matching the expert trajectory.

The objective changes from:

```text
Does the trajectory look like the expert trajectory?
```

to:

```text
Does executing this trajectory lead to good long-term driving outcomes?
```

Typical optimization goals include safety, comfort, efficiency, rule compliance, human-like behavior, long-tail scenario handling, and interactive decision quality.

## 3. Typical Pipeline

```text
Train a base policy π_BC with imitation learning
        ↓
Build closed-loop simulation / offline replay
        ↓
Define reward / cost / preference signals
        ↓
Generate candidate trajectories
        ↓
Run closed-loop rollout and evaluate outcomes
        ↓
Update policy with RL / Offline RL / Preference Optimization
        ↓
Constrain policy drift from π_BC
        ↓
Filter with safety rules and trajectory optimization
        ↓
Run large-scale simulation regression tests
        ↓
Validate with shadow mode and road tests
```

The key point is that fine-tuning should evaluate closed-loop outcomes, not only one-step trajectory similarity.

## 4. What Is Fine-Tuned

### 4.1 Direct Trajectory Output

The model directly outputs a future trajectory:

```text
a = τ = {(x1,y1), ..., (xT,yT)}
```

This is highly end-to-end, but the action space is high-dimensional and requires strict kinematic and safety checks.

### 4.2 Trajectory Scoring

A production-friendly approach is to generate multiple candidate trajectories and use a learned value or reward model to rank them.

```text
τ* = argmax Q(s, τ)
```

This is safer and easier to integrate with traditional planners because learning is used for ranking, while rules and optimizers still provide guardrails.

### 4.3 High-Level Behavior Selection

RL can also select high-level actions:

```text
keep lane / lane change / yield / creep / stop / accelerate / decelerate
```

The final trajectory is then generated by a conventional planner. This reduces action complexity and improves interpretability.

## 5. Closed-Loop Simulation

Open-loop evaluation compares predicted trajectories with expert trajectories, but the environment does not react to the model action.

Closed-loop evaluation executes the model trajectory, updates the ego state, lets other traffic participants react, and then asks the model to decide again.

Common simulation forms:

1. **Log replay**: cheap and realistic, but other agents may not react to changed ego behavior.
2. **Reactive simulation**: surrounding agents respond to ego behavior; useful for interaction-heavy cases.
3. **Generative / neural simulation**: learned traffic agents generate responses; useful for long-tail and complex scenarios.

## 6. Reward and Cost Design

A typical reward can be written as:

```text
R = w_safe R_safe
  + w_rule R_rule
  + w_comfort R_comfort
  + w_eff R_efficiency
  + w_human R_humanlike
  - β KL(π || π_BC)
```

### 6.1 Safety

Safety metrics include collision, time to collision, safe distance, collision probability, and minimum distance to vehicles, pedestrians, and cyclists. Collision is usually treated as a hard constraint.

### 6.2 Rule Compliance

Rules include no red-light running, no solid-line crossing, no wrong-way driving, no driving into non-drivable areas, no speeding, route following, and correct yielding at intersections. Many of these should be hard constraints.

### 6.3 Comfort

Comfort metrics include longitudinal acceleration, lateral acceleration, jerk, yaw rate, steering rate, and curvature change. The goal is to avoid harsh braking, sudden acceleration, aggressive steering, oscillation, and non-smooth trajectories.

### 6.4 Efficiency

Efficiency metrics include progress along the route, reasonable speed, avoiding unnecessary stops, smooth intersection crossing, and reasonable lane-change completion. The efficiency weight must be controlled to avoid overly aggressive behavior.

### 6.5 Human-Like Style

Human-like driving can be learned from expert trajectory distance, human preference labels, learned reward models, pairwise trajectory ranking, takeover data, and disengagement data.

## 7. KL Constraint to the Imitation Policy

RL fine-tuning needs to prevent the policy from drifting too far from the imitation-learned reference policy:

```text
R_total = R_driving - β D_KL(π || π_BC)
```

This prevents reward hacking, avoids strange trajectories, keeps the policy near the normal driving distribution, and improves production stability. It is similar to the reference-policy constraint used in RLHF for language models.

## 8. Common Fine-Tuning Methods

### 8.1 Simulation-Based RL

The policy interacts with a simulator, executes trajectories, receives rewards, and updates the policy. It can optimize long-term closed-loop behavior, but depends heavily on simulator fidelity and suffers from sim-to-real gap.

### 8.2 Offline RL

Offline RL learns from existing driving datasets without real-world exploration. It is more suitable for autonomous driving, but must handle out-of-distribution actions and value overestimation with conservative constraints.

### 8.3 Preference Learning / RLHF-Like Optimization

Humans, rules, or expert systems compare two trajectories:

```text
Trajectory A vs. Trajectory B: which one is better?
```

A reward model is trained:

```text
Rθ(s, τ)
```

Then the policy is optimized against this learned reward. This is useful for comfort, human-like style, aggressive versus conservative behavior, and product-specific driving style.

## 9. Production-Oriented Form

Production systems usually do not deploy an unconstrained RL policy directly to control the vehicle. A common form is:

```text
Imitation-learned policy generates candidate trajectories
        ↓
RL / reward model / value model ranks trajectories
        ↓
Rules and safety modules filter trajectories
        ↓
Trajectory optimizer smooths the selected trajectory
        ↓
Controller executes the trajectory
```

This combines learning capability, safety, interpretability, and engineering controllability.

## 10. Summary

RL-based trajectory fine-tuning after imitation learning means using imitation learning to acquire basic driving ability, then using RL, offline RL, or preference learning to improve long-term driving quality.

In production, RL is usually combined with safety rules, trajectory optimization, simulation validation, shadow mode, and data flywheels instead of being used as an unconstrained direct controller.

# 模仿学习后的 RL 轨迹微调

## 1. 背景

端到端自动驾驶通常先从模仿学习开始。基础策略学习如何把感知、导航和历史状态映射成自车未来轨迹。

```text
Input: sensors / BEV / Occupancy / map / navigation / history
Output: trajectory τ = {(x1,y1,v1,yaw1), ..., (xT,yT,vT,yawT)}
```

行为克隆的目标通常是：

```text
L_BC = ||τ_pred - τ_expert||
```

这能让模型获得基本驾驶能力，但也有局限：行为容易平均化、遇到分布外状态不稳定、只拟合短期轨迹、可能复制人类坏习惯，并且 open-loop 误差小不代表 closed-loop 驾驶质量好。

## 2. 为什么需要 RL 微调

模仿学习之后，可以用 RL、Offline RL 或 Preference Learning 来优化轨迹执行后的结果，而不仅是拟合专家轨迹。

目标从：

```text
这条轨迹像不像专家轨迹？
```

变为：

```text
执行这条轨迹后，长期驾驶结果好不好？
```

典型优化目标包括安全性、舒适性、通行效率、规则合规、类人驾驶、长尾场景处理和交互决策质量。

## 3. 典型流程

```text
Train a base policy π_BC with imitation learning
        ↓
Build closed-loop simulation / offline replay
        ↓
Define reward / cost / preference signals
        ↓
Generate candidate trajectories
        ↓
Run closed-loop rollout and evaluate outcomes
        ↓
Update policy with RL / Offline RL / Preference Optimization
        ↓
Constrain policy drift from π_BC
        ↓
Filter with safety rules and trajectory optimization
        ↓
Run large-scale simulation regression tests
        ↓
Validate with shadow mode and road tests
```

关键点是：微调应评估闭环执行结果，而不只是单步轨迹相似度。

## 4. 微调对象

### 4.1 直接输出轨迹

模型直接输出未来轨迹：

```text
a = τ = {(x1,y1), ..., (xT,yT)}
```

这种方式端到端程度高，但动作空间维度高，并且需要严格的运动学和安全检查。

### 4.2 轨迹打分

更适合量产的方式是先生成多条候选轨迹，再用学习到的 value model 或 reward model 对轨迹排序。

```text
τ* = argmax Q(s, τ)
```

这种方式更安全，也更容易和传统规划器结合，因为学习模型负责排序，而规则和优化器提供兜底。

### 4.3 高层行为选择

RL 也可以只选择高层动作：

```text
keep lane / lane change / yield / creep / stop / accelerate / decelerate
```

最终轨迹由传统 planner 生成。这可以降低动作复杂度，并提升可解释性。

## 5. 闭环仿真

Open-loop 评估只比较预测轨迹和专家轨迹，环境不会响应模型动作。

Closed-loop 评估会执行模型轨迹，更新自车状态，让其他交通参与者响应，然后让模型继续决策。

常见仿真形式：

1. **Log replay**：成本低且场景真实，但其他 agent 通常不会响应 ego 行为变化。
2. **Reactive simulation**：周围 agent 会响应自车行为，适合交互密集场景。
3. **Generative / neural simulation**：用学习型交通 agent 生成响应，适合长尾和复杂交互场景。

## 6. Reward 和 Cost 设计

典型 reward 可以写成：

```text
R = w_safe R_safe
  + w_rule R_rule
  + w_comfort R_comfort
  + w_eff R_efficiency
  + w_human R_humanlike
  - β KL(π || π_BC)
```

### 6.1 安全性

安全指标包括碰撞、TTC、安全距离、碰撞概率，以及与车辆、行人、骑行者的最小距离。碰撞通常应作为硬约束。

### 6.2 规则合规

规则包括不闯红灯、不压实线、不逆行、不进入不可行驶区域、不超速、按路线行驶和路口正确让行。很多规则应作为硬约束。

### 6.3 舒适性

舒适性指标包括纵向加速度、横向加速度、jerk、yaw rate、steering rate 和曲率变化。目标是避免急刹、急加速、猛打方向、左右摇摆和不平滑轨迹。

### 6.4 效率

效率指标包括沿路线前进、合理速度、避免无意义停车、平顺通过路口和合理完成变道。效率权重需要谨慎控制，避免策略过于激进。

### 6.5 类人驾驶风格

类人驾驶可以从专家轨迹距离、人类偏好标注、学习型 reward model、轨迹 pairwise ranking、接管数据和 disengagement 数据中学习。

## 7. 相对模仿策略的 KL 约束

RL 微调需要防止策略偏离模仿学习得到的参考策略太远：

```text
R_total = R_driving - β D_KL(π || π_BC)
```

这有助于防止 reward hacking、避免奇怪轨迹、让策略保持在正常驾驶分布附近，并提升量产稳定性。这类似于大语言模型 RLHF 中的 reference policy 约束。

## 8. 常见微调方法

### 8.1 基于仿真的 RL

策略与仿真器交互，执行轨迹，获得 reward，并更新策略。它可以优化长期闭环表现，但高度依赖仿真真实性，也存在 sim-to-real gap。

### 8.2 Offline RL

Offline RL 从已有驾驶数据中学习，不进行真实道路探索。它更适合自动驾驶，但需要用保守约束处理分布外动作和价值高估问题。

### 8.3 Preference Learning / 类 RLHF 优化

人类、规则系统或专家系统比较两条轨迹：

```text
Trajectory A vs. Trajectory B: which one is better?
```

训练 reward model：

```text
Rθ(s, τ)
```

然后用该 reward 优化策略。这适合调节舒适性、类人驾驶风格、激进或保守程度，以及产品化驾驶风格。

## 9. 面向量产的常见形态

量产系统通常不会直接部署一个无约束的 RL policy 控车。更常见的形式是：

```text
Imitation-learned policy generates candidate trajectories
        ↓
RL / reward model / value model ranks trajectories
        ↓
Rules and safety modules filter trajectories
        ↓
Trajectory optimizer smooths the selected trajectory
        ↓
Controller executes the trajectory
```

这种方式兼顾学习能力、安全性、可解释性和工程可控性。

## 10. 总结

模仿学习后的 RL 轨迹微调，是指先用模仿学习获得基础驾驶能力，再用 RL、Offline RL 或 Preference Learning 优化长期驾驶质量。

在量产中，RL 通常会和安全规则、轨迹优化器、仿真验证、shadow mode 和数据闭环结合使用，而不是作为无约束的直接控制器。
