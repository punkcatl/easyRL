# On-Policy vs Off-Policy

## Core Difference

- **On-policy**: Train using data generated by the current policy. Data is discarded after each update.
- **Off-policy**: Train using data from any source (old policy, replay buffer, even human demonstrations). Data is stored and reused.

## Comparison

| | On-Policy (PPO) | Off-Policy (SAC/DQN) |
|---|---|---|
| Data usage | Use once then discard | Store in buffer, reuse many times |
| Sample efficiency | Low (wasteful) | High (efficient) |
| Stability | More stable, less likely to diverge | Prone to divergence, Q-value overestimation |
| Implementation | Simple | Complex (target network, replay buffer, fixes) |
| Convergence guarantee | Stronger theoretical guarantee | Weaker, requires careful engineering |

## Why Off-Policy Is Hard: Distribution Shift

```
Buffer stores data from old policy π_old
Currently updating new policy π_new
Old data may be "wrong experience" for the new policy
```

Example:
- Old policy crashes often → buffer full of crash data
- New policy has learned to avoid obstacles
- Still learning from "crash experience" → Q-value overestimation → policy collapse

Fixes required for off-policy:
- Target network (stabilize Q targets)
- Double Q (reduce overestimation)
- Large buffer + random sampling (break correlation)
- Various clip/regularization tricks

On-policy has none of these problems — data always comes from the current policy, so gradient direction is always correct.

## When to Choose Which

| Scenario | Choice | Reason |
|----------|--------|--------|
| Cheap environment interaction (simulator) | On-policy (PPO) | Stable, less tuning |
| Expensive interaction (real robot) | Off-policy (SAC) | Can't waste data |
| Want quick results, minimal tuning | PPO | Works out of the box |
| Continuous control, need sample efficiency | SAC | Buffer reuses data |

## Summary

Off-policy trades complexity for efficiency; on-policy trades data waste for stability. Neither is strictly better — it depends on the cost of environment interaction and tolerance for implementation complexity.

# On-Policy 与 Off-Policy 对比

## 核心区别

- **On-policy**：用当前策略生成的数据训练，数据用完就扔。
- **Off-policy**：可以用任意来源的数据训练（旧策略、replay buffer、甚至人类示范），数据存起来反复用。

## 对比

| | On-Policy (PPO) | Off-Policy (SAC/DQN) |
|---|---|---|
| 数据利用 | 用完就扔，浪费 | 存 buffer 反复用，高效 |
| 样本效率 | 低 | 高 |
| 稳定性 | 更稳定，不易崩 | 容易发散、Q 值高估 |
| 实现复杂度 | 简单 | 需要目标网络、replay buffer、各种修补 |
| 收敛保证 | 理论保证更强 | 理论保证弱，实践中要小心 |

## Off-Policy 为什么难：分布偏移（Distribution Shift）

```
Buffer 里存的是旧策略 π_old 产生的数据
当前要更新的是新策略 π_new
旧数据对新策略来说可能是"错误的经验"
```

举例：
- 旧策略经常撞车，buffer 里全是撞车数据
- 新策略已经学会避障了，但还在从"撞车经验"里学
- 如果处理不好 → Q 值高估 → 策略崩塌

Off-policy 需要的修补措施：
- 目标网络（稳定 Q 值目标）
- Double Q（减少高估）
- 大 buffer + 随机采样（打破相关性）
- 各种 clip / 正则化

On-policy 天然没有这个问题——数据就是当前策略产生的，梯度方向一定是对的。代价是数据用完就扔。

## 选择依据

| 场景 | 选哪个 | 原因 |
|------|--------|------|
| 和环境交互很便宜（模拟器） | On-policy (PPO) | 稳定，调参少 |
| 和环境交互很贵（真实机器人） | Off-policy (SAC) | 不能浪费数据 |
| 需要快速出结果、不想调参 | PPO | 开箱即用 |
| 连续控制、需要高样本效率 | SAC | buffer 复用数据 |

## 总结

Off-policy 用复杂度换效率，On-policy 用数据浪费换稳定性。没有绝对更好——取决于环境交互成本和对实现复杂度的容忍度。
